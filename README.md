# Tabs – Localization Automation Project

A work-in-progress localization pipeline built around **Tabs**, a mock mental clarity app.

🌐 Live site: [tabs.egle.works](https://tabs.egle.works)

---

## Why this exists

Started to explore what current AI automation capabilities actually look like in practice.

---

## What I'm figuring out

AI can write code, generate scripts, guide integrations step by step, and help optimize prompts. But it has tunnel vision. It needs the right questions asked and real-world constraints pointed out.

One example: the default setup had strings managed directly in GitHub – which no real localization workflow does. Copywriters work in Google Sheets, not repos. Another example: Crowdin free tier has no webhook support, so n8n uses scheduled polling instead. Every constraint like this shapes the pipeline.
---

## Pipeline (in progress)

```
Google Sheets (CW copy source)
        ↓
n8n → generates en.json → pushes to GitHub
        ↓
Crowdin picks up changes via GitHub integration
        ↓
DE/JA: MT pretranslation in Crowdin (manual trigger on free tier)
        ↓
n8n polls Crowdin → AI quality review → results logged to review sheet
        ↓
Human reviews flagged strings → approves in Crowdin → pushed to GitHub

LT separate path:
n8n reads untranslated strings from Crowdin → AI translation (Groq) 
→ AI quality review → results logged to review sheet
→ Human reviews flagged strings → approved translations pushed to GitHub

```

Copy is managed in a Google Sheet (the single source of truth for all English strings). An n8n workflow syncs the sheet to locales/en/en.json in this repository via the GitHub API on a daily trigger – no manual file editing is required. Crowdin detects the commit and pulls the updated source strings automatically.

DE and JA translations are produced via MT pre-translation in Crowdin (triggered manually on free tier). LT translations are generated by an n8n AI translation workflow (Groq/Llama) with glossary and style guide injection, bypassing Crowdin MT entirely. All translations go through an n8n AI quality review workflow (MQM-style evaluation) with results logged to a dedicated review sheet. A human reviewer checks flagged strings, fixes them in Crowdin, and approves before translations are pushed back to GitHub.

Translation routing is driven by a language_config sheet – adding a new language and assigning it a translation method (crowdin_mt or ai_translate) is all that's needed to extend the pipeline.

The landing page at tabs.egle.works is currently updated manually from GitHub; auto-deploy via Netlify is planned as the next step.
